## Milestone Summary

Milestone M0 established the project's foundation: a version-controlled, installable, properly
tooled Python package skeleton that runs end to end before any feature code exists. The goal was
explicitly *not* to build functionality but to remove every future source of friction —
packaging, imports, tooling, workflow — so that later milestones add features into a clean,
predictable structure. M0 is **Completed and merged**, including two post-audit documentation
follow-up PRs.

---

## Goals

The milestone aimed to: create the repository and enforce a professional Git workflow; lay down
the full folder and package structure expressing the layered architecture; configure dependency
management and the quality toolchain in a single canonical file; establish an isolated Python
environment with an editable install; and provide a minimal runnable entry point that proves the
package executes. A successful M0 means a new developer can clone, install, and run the project
without guesswork.

---

## Tasks Completed

The milestone corresponds to Issues #1–#8 plus two post-audit documentation PRs. Work completed:
repository creation and clone; branch protection on `main`; pull-request workflow establishment;
authoring `.gitignore` and `.env.example`; creation of the full `src/app/...` package tree, the

## Repository Initialization

The repository was created on GitHub and cloned locally, with the first commit established as a
scaffold commit rather than feature code. The project is public. The canonical project definition
was placed in `pyproject.toml` from the outset rather than accreting configuration files later.

---

## GitHub Setup

The repository uses GitHub as the remote of record. A pull-request-based workflow was adopted: all
changes land on `main` through PRs rather than direct pushes. Conventional Commits
(`feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`) are the commit message standard, chosen
to teach disciplined history and to enable future changelog automation.

---

## Branch Protection

Branch protection was enabled on `main` to require pull requests before merging and to block direct
pushes. The intent is to make the always-working-`main` rule mechanical rather than reliant on
discipline. **Assumption:** the precise protection ruleset (e.g. required reviews, required status
checks) was not independently re-verified for this report; the development record states protection
is enabled and the PR workflow is in force.

---

## Folder Structure

The structure physically expresses the architecture. Top level: `src/app/` (the package),
`tests/` (mirror of the source tree), `data/` (runtime files), `docs/` (documentation), and root
project files. Within `src/app/`: `config/`, `models/`, `services/`, `clients/`, `storage/`,
`presentation/`, and `utils/` — one sub-package per architectural concern. The `data/` tree
contains `cache/`, `settings/`, `history/`, and `logs/`, each holding a `.gitkeep` so the folder is
tracked while its contents are gitignored.

```
crypto-worldcup-platform/
├── src/app/
│   ├── main.py
│   ├── config/        models/        services/
│   ├── clients/       storage/       presentation/
│   └── utils/
├── tests/
├── data/  (cache/ settings/ history/ logs/  — .gitkeep tracked, contents ignored)
├── docs/
├── pyproject.toml  .gitignore  .env.example  .python-version
└── README.md  LICENSE  CHANGELOG.md  CONTRIBUTING.md
```

---

## Package Structure

The project uses a `src/` layout: the importable package is `app`, located under `src/`. Each
sub-package carries an `__init__.py` (empty placeholders at this stage). The `src/` layout was
chosen deliberately over a root-level flat package because it prevents importing the un-installed
package by accident — code must be installed (editable) to import it, which surfaces packaging
problems early and mirrors how the package behaves when distributed.

---

## Tooling Setup

Three tools are configured in `pyproject.toml`: **Ruff** (linting and formatting, replacing
flake8/black/isort with one fast tool), **mypy** (static type checking in strict mode), and
**pytest** (testing). The pre-merge quality gate is `ruff check .`, `ruff format .`, `mypy src`,
and `pytest`. The line length standard is 88. Strict mypy makes type hints load-bearing rather
than decorative.

---

## Dependency Management

`pyproject.toml` is the single source of truth. Runtime dependencies: `requests` (HTTP), `rich`
(terminal formatting), and `python-dotenv` (`.env` loading). Development dependencies live in a
`dev` optional-dependency group: `pytest`, `ruff`, `mypy`, and `types-requests`. A separate
hand-maintained `requirements.txt` was deliberately not adopted; `pyproject.toml` carries the
truth. The package is installed editable so source edits take effect without reinstalling.

---

## Virtual Environment Setup

Development uses a project-local virtual environment (`.venv`), with `python -m venv`, an upgraded
`pip`, and `pip install -e ".[dev]"` to install the package plus dev tooling. A `.python-version`
file pins the interpreter to 3.12. The README documents the activation commands per operating
system so onboarding is reproducible.

---

## Entry Point Setup

A console entry point, `crypto-wc`, is declared in `pyproject.toml` and mapped to the package's
`main` function. At the end of M0 this entry point ran a minimal `rich` banner and exited cleanly —
a deliberate smoke test proving the entire chain (install → entry point → package import →
execution) works before any feature logic is added. (The entry point was subsequently expanded in
M1 to wire the foundation; see the M1 report.)

---

## Testing and Validation

M0 validation was manual and structural rather than automated (the pytest *suite* is planned for
M7). Validation performed: confirming the editable install succeeds on a clean environment;
confirming `crypto-wc` runs and exits cleanly; and confirming the package imports. The governance
audit additionally verified the committed `pyproject.toml` against the approved plan and confirmed
the top-level structure. **Assumption:** no automated tests existed at M0 by design; the first
tests arrive in M7.

---

## Challenges

The principal issue surfaced during the post-scaffolding governance audit, which found two
documentation gaps that the scaffolding had not closed: the approved architecture document had not
yet been committed to `docs/` (finding C-1), and the README still read "Phase 0 in progress" with no
setup instructions (finding C-2). A structural inconsistency was also reconciled during planning:
because `data/` contents are gitignored and Git does not track empty directories, the runtime
folders would be absent on a fresh clone — resolved by tracking the skeleton with `.gitkeep` and
deferring directory creation to application startup (implemented in M1; see ADR-012). Branch
protection and PR history could not be inspected by automated tooling during the audit and were
taken from the development record.

---

## Lessons Learned

Three lessons stand out. First, scaffolding audits tend to fail on documentation, not code — the
two Critical findings were both docs, which is typical because there is barely any code yet to be
wrong. Second, the gitignored-empty-directory problem is a classic onboarding bug that only appears
for the *second* person to clone the repo; catching it during planning saved a confusing first-run
failure later. Third, committing the architecture document into the repository (rather than leaving
it in planning notes) is what makes it a living source of truth that future PRs can update.

## Deliverables

A cloneable, installable, runnable package: full `src/app` layered structure with `tests/` mirror;
`data/` skeleton tracked via `.gitkeep`; `pyproject.toml` with metadata, dependencies, and Ruff /
mypy / pytest configuration; `.gitignore`, `.env.example`, `.python-version`; root documentation
(`README.md`, `LICENSE`, `CONTRIBUTING.md`, `CHANGELOG.md`); a working `crypto-wc` entry point;
and a committed `docs/architecture.md`.

---

## Future Dependencies

Everything built later depends on M0. Specifically: the M1 foundation modules live in the
`utils/` and `config/` packages created here; the `data/` skeleton and the `.gitkeep`/startup-
creation decision are consumed by the M1 settings loader and the M3 storage layer; the toolchain
configured here is the quality gate for every subsequent PR; and the Git workflow established here
governs how all future issues are merged. M0 created no functional behavior, but every functional
milestone is built on its structure.

## Milestone Summary

Milestone M1 built the cross-cutting foundation that every higher layer depends on: a custom
exception hierarchy, centralized logging, and an immutable configuration loader — then integrated
the three into a deliberately ordered application startup sequence in `main.py`. The defining
characteristic of M1 is that none of it is a feature the user sees directly; all of it is
infrastructure that makes later features safe to build. M1 is **Completed and merged**, verified by
REPL checks and startup smoke tests.

To implement the utility and configuration layers of the architecture: a single error vocabulary

## Exception Hierarchy

**What:** A single base class, `AppError`, derived from `Exception`, with three subclasses:
`ConfigError`, `StorageError`, and `APIError`. The subclasses are intentionally minimal —
docstring-only, with no custom fields or `__init__`.

**Why:** The governing rule is *exception translation at boundaries*. Each layer catches the
foreign exceptions it encounters (configuration parsing errors, `json`/`OSError` from storage,
`requests` errors from clients) and re-raises them as the appropriate application type. This
prevents lower-level library details from leaking upward — for example, the service layer never
catches `requests.ConnectionError`, because doing so would couple it to the fact that a client uses
`requests`, defeating the adapter pattern (ADR-003).

**Why a hierarchy rather than a flat list:** a hierarchy lets callers choose their precision. Code
can catch one specific type, an entire family (`StorageError`), or every anticipated application
error via the base `AppError`. The single base also enables the top level to handle *anticipated*
failures gracefully while letting genuinely unexpected exceptions (a real bug, e.g. a `KeyError`)
surface loudly instead of being silently swallowed.

**Why minimal subclasses:** adding fields like `status_code` now would be speculative. Extension is
deferred until a real caller needs it (anticipated in M4 for `APIError`), per the project's
build-the-seam-not-the-feature principle.

**Verification:** Liskov substitution was confirmed in a REPL — a subclass instance is caught by
`except AppError`.

## Configuration Layer

**What:** `config/settings.py` defines `Settings`, a `@dataclass(frozen=True)` holding `data_dir`,
`cache_ttl_seconds`, `crypto_api_key`, and `football_api_key`. A `from_env` classmethod builds it
from the environment: it calls `load_dotenv()`, reads variables with defaults
(`DATA_DIR` → `data`, `CACHE_TTL_SECONDS` → `300`), validates them, and translates malformed values
into `ConfigError` with exception chaining (`from exc`). Subdirectory paths (`cache_dir`,
`settings_dir`, `history_dir`, `logs_dir`) are `@property` values derived from `data_dir`. An
`ensure_directories` method creates all runtime folders idempotently via `mkdir(parents=True,
exist_ok=True)`.

**Why immutable:** configuration is read-only after load; freezing it prevents a class of bug where
some module mutates shared configuration mid-run.

**Why centralized:** nothing except `settings.py` reads the environment. This makes it possible to
answer "what configuration is this run using?" from one object, and it makes testing tractable —
inject a `Settings` instance rather than monkeypatching global environment state.

**Why derived subpaths:** there is one source of truth (`data_dir`); the layout is computed from it,
so changing the root via `DATA_DIR` moves every subdirectory consistently. Storing the subpaths as
independent fields would let them drift out of sync.

**Why API keys default to empty:** the keys are introduced in M4. Requiring them now would block the
app from running through M1–M3 for no benefit. Validation is therefore deferred to the point of use
— the M4 clients will require their key before making a call.

**Verification:** REPL confirmed that reassigning a field raises `FrozenInstanceError`, and that a
malformed `CACHE_TTL_SECONDS` raises `ConfigError` (not a raw `ValueError`).

## Logging Layer

**What:** `utils/logger.py` exposes `setup_logging(log_dir, *, console_level=WARNING,
file_level=DEBUG)` and `get_logger(name)`. `setup_logging` configures the **root** logger once,
setting its level to DEBUG (a permissive gate) and attaching two handlers: a
`RotatingFileHandler` writing to `<log_dir>/app.log` at DEBUG and up (≈1 MB rollover, 3 backups,
verbose format), and a `StreamHandler` console handler at WARNING and up (terse format). It clears
existing handlers first, making repeated calls idempotent. `get_logger` is a thin wrapper over
`logging.getLogger`.

**Why configure once, get everywhere:** the professional pattern separates *getting* a logger (done
in every module via `getLogger(__name__)`) from *configuring* logging (done once at startup). Per-
module `basicConfig` produces duplicated handlers, conflicting formats, and double-printed lines.

**Why the root is DEBUG and the handlers filter:** the logger's level is a gate the record clears
*before* handlers see it. Setting the root permissive and letting each handler apply its own level
is what allows the file to capture DEBUG while the console stays quiet at WARNING. Setting the root
to WARNING would starve the file handler of DEBUG records regardless of its own level — a common
beginner mistake.

**Why the idempotency guard:** without `handlers.clear()`, calling `setup_logging` twice (in tests,
or if startup ever runs more than once) attaches a second handler pair and every line prints twice.

**Why lazy `%`-formatting:** logging calls use `logger.info("x=%s", v)`, not f-strings, so
interpolation only happens if the record is actually emitted.

**Verification:** a startup INFO message appears in `data/logs/app.log` but not on the console,
confirming the dual-handler level routing.

## Architecture Decisions

M1 implemented or exercised the following decisions from the log: ADR-009 (custom exception
hierarchy with boundary translation), ADR-010 (centralized dual-handler logging), ADR-011
(centralized immutable environment-sourced configuration), and ADR-012 (runtime directories created
at startup). It also exercised ADR-001's dependency direction: all three foundation modules sit in
the utility/config layers and depend only downward (on the standard library, `python-dotenv`, and —
for `settings.py` — `utils.exceptions`). See [`decisions.md`](../decisions.md) for full entries.

## Data Flow

M1's data flow is the **startup sequence** in `main.py`, whose ordering is significant:

1. **Load settings** — `Settings.from_env()`. On `ConfigError`, print the message to **stderr** and
   exit with status 1. (Logging is not yet configured at this point, so the failure cannot be
   logged — it is printed. This is a genuine ordering constraint, not an oversight.) Only
   `ConfigError` is caught; unexpected exceptions are left to surface.
2. **Ensure directories** — `settings.ensure_directories()`, before logging writes anything.
3. **Configure logging** — `setup_logging(settings.logs_dir)`.
4. **Emit startup log** — `get_logger(__name__).info(...)` with lazy `%`-formatting.
5. **Show the banner** — the `rich` panel, only after a clean start.

`main.py` is the composition root — the one place permitted to import across multiple layers and
wire them together. This is correct for an entry point, not a layering violation.

---

## Error Handling Strategy

The strategy has three parts. **Translate at boundaries:** foreign exceptions become `ConfigError` /
`StorageError` / `APIError` at the layer that encounters them. **Preserve cause:** translation uses
`from exc` so tracebacks still show the original error during debugging. **Fail loud but clean for
anticipated errors, crash loudly for bugs:** the top level catches the anticipated
`ConfigError` and exits with a clear stderr message and a non-zero status, while deliberately not
wrapping everything in a blanket `except` — so an unanticipated exception still produces a full
traceback rather than being masked. The asymmetry is intentional: `AppError` means "we anticipated
this"; everything else means "this is a bug, show it."

## Known Limitations

The default `log_dir` and `data_dir` are relative to the current working directory, so behavior
depends on launching from the project root; this is acceptable for V1 and will be tightened when an
explicit resolved path is passed. `load_dotenv()` does nothing if no `.env` file is present, so a
developer who skips copying `.env.example` gets empty keys rather than a warning (covered by the
README setup step). API keys are unvalidated until first use in M4. There is no automated test
coverage yet. None of these is a defect; each is a conscious V1 boundary.

## Lessons Learned

The most transferable lesson is that *ordering is a design concern*: the fact that configuration
failures cannot be logged (because logging is configured after settings load) forces a deliberate
stderr fallback and shapes the whole startup sequence. A second lesson is the value of the
permissive-root / filtering-handlers logging model — getting that relationship backwards is a
common and confusing mistake, and seeing the INFO-in-file-but-not-on-console behavior makes it
concrete. A third is that immutability and centralization in configuration pay off immediately in
testability, not just in some distant future.

## Review Notes

Against the project's code review checklist, the M1 work holds: layer boundaries respected
(foundation modules depend only downward); full type hints with strict mypy; specific exceptions
rather than bare `except`; no secrets logged; meaningful names; no unjustified dependencies
(`python-dotenv` was pre-approved; `rich` is deliberately kept out of the logging layer to preserve
the UI/utility boundary). Two seams were consciously accepted and documented rather than hidden: the
relative default paths and the deferred API-key validation. The foundation is considered solid
enough to build the domain and storage layers on top of without revisiting.
